Реестр узлов ?

Локальный реестр работает вместе с синхронизацией из Marzban: Marzban остаётся источником истины о существовании и внешнем source-статусе узла, а проект хранит локальные бизнес-поля — приоритеты, теги, заметки, default-узел и публичные базовые URL.

Web-admin работает в локальном режиме и должен открываться через SSH-туннель.

Создать / обновить узел ?

Если указать уже существующий code, форма обновит существующую запись. Для новой записи обязательно укажите code и display_name.

Важно по nullable-полям

Пустые значения в API base URL и Subscription base URL реально очищают поле в БД. Старое значение не сохраняется молча.

Что управляется источником, а что локально
  • Из Marzban: существование узла, внешний source id, source status и часть диагностического payload.
  • Локально: display name, priority, weight, tags, default, notes, публичные base URL и ручной health status.

Памятка по синхронизации и policy routing ?

Здесь мы описываем узлы и policy tags, а source-состояние синхронизируем из Marzban без включения случайной multi-node маршрутизации в боевом потоке.

Как читать sync-статусы
  • synced — узел найден в Marzban и успешно сопоставлен с локальной записью.
  • missing — локальная запись есть, но узел больше не пришёл из источника.
  • error — синхронизация завершилась ошибкой и требует проверки.
  • never_synced — локальная запись ещё ни разу не была подтверждена из Marzban.
Рекомендуемые policy tags
  • ai-clean — foundation под чистые egress-маршруты для AI-сервисов.
  • ru-bridge — foundation под bridge-профили и обходные сценарии.
  • mobile-stable — консервативные профили под мобильных клиентов.
  • low-risk-reality — осторожные reality-профили без случайной маршрутизации.
Не включайте здесь WARP как основу маршрутизации и не используйте случайную multi-node маршрутизацию для SelfSNI на этом этапе.
